### **비즈니스 모델: Sentinel (센티넬): The Predictive Health OS**

**한 줄 정의:** 매일 사용하는 화장실을 개인 맞춤형 건강 연구소로 전환하여, 질병을 '예측'하고 '예방'하는 구독형 헬스케어 플랫폼입니다.

### **타겟 페인 포인트 (The Broken System)**

현재의 헬스케어는 근본적으로 잘못됐습니다. 우리는 '아프고 나서야' 병원에 갑니다. 이미 증상이 발현된 후죠. 시간, 돈, 감정 모든 것이 소모되는 '사후약방문(Reactive)' 모델입니다. 스마트 워치가 걸음 수를 세어주는 건 좋지만, 그건 건강의 '표면'일 뿐입니다. 암, 당뇨, 치매 같은 치명적인 질병의 '전조증상'은 몸 속 깊은 곳, 즉 혈액, 소변, 대변 같은 생체 데이터에 숨어있습니다. 하지만 이걸 매일 검사하는 건 불가능하죠. 바로 이 '데이터 공백'이 우리가 해결할 가장 고통스러운 지점입니다.

### **우리의 솔루션 (The Disruption)**

우리는 값비싼 스마트 토일렛을 파는 게 아닙니다. 기존의 어떤 변기에도 10분 안에 설치 가능한 '센서 모듈'을 제공합니다. 이 모듈은 사용자가 인지하지도 못하는 사이, 매일 배출되는 소변과 대변을 분석하여 핵심 바이오마커(Biomarker) 데이터를 추출합니다.

1.  **하드웨어 (The Data Collector):** 변기 내부에 부착하는 소형 센서킷. 분광학, 화학 센서 등을 이용해 소변의 포도당, 케톤, 단백질 수치, 대변의 잠혈 반응, 장내 미생물군(마이크로바이옴) 변화 등 수십 가지 데이터를 매일 자동으로 수집합니다. 가족 구성원은 앱을 통해 간단히 등록하여 개인별로 데이터를 관리합니다.
2.  **소프트웨어 (The AI Brain):** 수집된 데이터는 클라우드로 전송되어 우리의 AI 엔진이 분석합니다. 단기적인 변화는 물론, 수개월에 걸친 장기적인 데이터 패턴을 분석하여 질병의 '전조증상'을 포착합니다. "지난 3주간 특정 염증 수치가 미세하게 상승 중입니다. 대장 내시경을 권장합니다" 와 같은 '실행 가능한(Actionable) 인사이트'를 앱을 통해 제공합니다.

### **수익화 전략 (The Unbeatable Economics)**

이 모델의 LTV(고객생애가치)는 CAC(고객획득비용)를 압도합니다.

1.  **B2C 구독 (기본 엔진):**
    *   **Standard ($19/월):** 개인 건강 리포트 및 이상 징후 알림.
    *   **Premium ($49/월):** 가족 단위 관리, AI 기반 식단 및 운동 추천, 원격 의료 파트너 연계.

2.  **B2B2C 보험사 파트너십 (성장 가속기):** 보험사에게 이보다 더 좋은 딜은 없습니다. 고객의 암을 조기에 발견하면 수억 원의 보험금 지출을 막을 수 있습니다. 보험사가 고객에게 우리 센서 모듈 설치를 보조해주고, 우리는 그들에게 익명화된 리스크 관리 데이터를 제공합니다. 고객은 보험료 할인, 보험사는 비용 절감, 우리는 시장 지배력 확보. 모두가 윈윈하는 구조입니다.

3.  **데이터 라이선싱 (The Real Endgame):** 수백만 명으로부터 매일 수집되는 전례 없는 규모의 '장기 추적(Longitudinal) 생체 데이터'는 그 자체로 21세기의 원유입니다. 제약사, 신약 개발 연구소에 이 익명화된 데이터를 판매하여 신약 개발 사이클을 획기적으로 단축시킬 수 있습니다. 이것이 우리가 조 단위 가치를 넘어설 핵심 동력입니다.

### **5년 뒤의 미래 시나리오 (The New World)**

5년 뒤, '센티넬'은 가정의 필수품이 되었습니다.

*   의사와의 대화가 바뀝니다. "어디가 아프세요?"가 아니라, "지난 3개월간의 센티넬 데이터를 보니, 신장 기능 관련 수치가 좋지 않군요. 정밀 검사를 해봅시다." 로 시작합니다.
*   '국민 건강검진'의 패러다임이 바뀝니다. 1~2년에 한 번 받는 형식적인 검진이 아니라, 매일의 데이터를 기반으로 한 '상시적 건강관리' 시대가 열립니다.
*   우리는 더 이상 '아프지 않은 상태'를 건강이라 부르지 않습니다. 최적의 건강 상태를 '설계'하고 '유지'하는 시대를 여는 겁니다.

이것은 단순한 헬스케어 기기가 아닙니다. 가정에 설치된 '데이터 OS'이자, 인류의 수명을 연장시키는 거대한 플랫폼의 시작입니다. 기술적 디테일은 아키텍트와 논의하면 됩니다. 중요한 건, 이 비즈니스가 인류의 가장 근본적인 문제인 '건강과 수명'을 정면으로 다루고 있다는 사실입니다.

이 정도 스케일이 아니면 '유니콘'이라는 단어를 꺼내선 안되겠죠. 어떻게 생각하십니까?

[디지털 아키텍트]:
훌륭한 비전입니다. 시장을 점진적으로 개선하는 것이 아니라, '예방적 헬스케어(Proactive Healthcare)'라는 새로운 카테고리를 창출하려는 야심이 느껴집니다. 이는 단순한 앱 개발이 아닌, 하드웨어, 소프트웨어, 그리고 인류의 행동 패턴까지 바꾸는 거대한 시스템 엔지니어링 프로젝트입니다.

도메인 전문가의 비전을 기술적 관점으로 재해석하고, 이 거대한 구상을 실행 가능한 청사진으로 설계해 보겠습니다.

### **기술적 관점에서의 재해석 (Re-interpretation)**

'Sentinel'의 본질은 **'고밀도 시계열 생체 데이터(High-Density Time-Series Biometric Data)를 수집하여 개인화된 질병 확률 모델을 구축하는 분산 시스템'**입니다.

- **The Predictive Health OS:** 이는 클라우드 기반의 MLOps(Machine Learning Operations) 플랫폼입니다. 핵심은 데이터 수집(IoT), 처리(Data Pipeline), 분석(ML Model), 그리고 실행(Mobile App)의 선순환 구조를 자동화하는 것입니다.
- **센서 모듈:** Edge Computing 디바이스에 해당합니다. 원시 데이터를 수집하고, 최소한의 전처리 후 클라우드로 안전하게 전송하는 역할을 맡습니다.
- **수익 모델:** 기술적으로는 API 기반의 서비스형 플랫폼(PaaS, Platform-as-a-Service)으로 구현됩니다. B2C 앱, B2B2C 파트너, B2B 데이터 소비자는 모두 이 플랫폼의 API를 통해 각기 다른 수준의 데이터와 인사이트에 접근하게 됩니다.

### **시스템 아키텍처 (System Architecture)**

100만 유저의 일일 데이터를 처리하고, 실시간에 가까운 분석을 제공하며, 의료 데이터 수준의 보안을 유지하기 위해 Microservice Architecture 기반의 이벤트 주도형(Event-Driven) 설계를 제안합니다.

### **추천 기술 스택 (Tech Stack)**

- **Frontend (Mobile App):**
  - **React Native / Flutter:** 단일 코드베이스로 iOS와 Android를 동시에 지원하여 MVP 개발 속도를 극대화합니다. 블루투스 통신, 푸시 알림 등 네이티브 기능 접근이 용이한 React Native를 우선 추천합니다.

- **Backend (Microservices):**
  - **언어:** **Python & Go.** 데이터 처리와 AI/ML 모델 서빙에는 Python(FastAPI, Pandas, Scikit-learn)이 절대적으로 유리합니다. 대량의 동시 요청을 처리하는 데이터 수신(Ingestion) 서비스는 Go(Gin)를 사용하여 성능을 극대화합니다.
  - **인프라:** **Docker & Kubernetes (AWS EKS).** 서비스의 독립적인 배포, 확장, 관리를 위한 표준입니다.

- **Database:**
  - **시계열 데이터베이스 (Time-Series DB):** **TimescaleDB** 또는 **Amazon Timestream.** 수십억 건의 바이오마커 데이터를 시간 순으로 저장하고, 특정 기간의 트렌드 분석 쿼리를 초고속으로 처리하는 데 최적화되어 있습니다. 일반 RDBMS로는 확장성 한계에 부딪힙니다.
  - **사용자 메타 데이터베이스:** **Amazon Aurora (PostgreSQL 호환).** 사용자 프로필, 구독 정보, 가족 관계 등 정형 데이터를 안정적으로 관리합니다.
  - **데이터 레이크 (Data Lake):** **Amazon S3.** 모든 원시(Raw) 데이터를 비용 효율적으로 저장합니다. 향후 새로운 AI 모델을 학습시키기 위한 가장 중요한 자산입니다.

- **AI Model & MLOps:**
  - **플랫폼:** **Amazon SageMaker** 또는 **Kubeflow.** 모델 개발, 학습, 배포, 모니터링의 전 과정을 자동화합니다.
  - **MVP 모델:** 초기에는 통계 기반의 **규칙 기반 엔진(Rule-based Engine)**으로 시작합니다. ("3일 연속 케톤 수치 '높음' 시 경고")
  - **고도화 모델:** 데이터가 축적되면, 개인별 정상 범위를 학습하는 **이상 탐지(Anomaly Detection) 모델(LSTM, Transformer 기반)**과 특정 질병의 전조 패턴을 학습하는 **분류(Classification) 모델**로 발전시킵니다.

### **데이터 흐름 (Data Flow)**

1.  **수집 (Collect):** 센서 모듈이 바이오마커 데이터를 측정하고, 블루투스를 통해 사용자의 스마트폰 앱으로 전송합니다. 앱은 가족 구성원을 식별(예: 사용 전 앱에서 사용자 선택)하고 데이터를 1차 취합합니다.
2.  **전송 (Ingest):** 앱은 HTTPS 프로토콜을 통해 암호화된 데이터를 AWS API Gateway로 전송합니다.
3.  **분리 및 저장 (Process & Store):** 데이터 수신 서비스는 요청을 받아 메시지 큐(Kafka/SQS)에 적재합니다. 데이터 처리 파이프라인은 큐의 메시지를 비동기적으로 처리하여, 원시 데이터는 S3 데이터 레이크에, 정제된 시계열 데이터는 TimescaleDB에, 관련 메타데이터는 Aurora에 저장합니다.
4.  **분석 (Analyze):** 새로운 데이터가 저장되면 이벤트가 발생하여 실시간 분석 엔진과 ML 추론 파이프라인을 트리거합니다. 시스템은 단기적 이상치와 장기적 패턴 변화를 동시에 분석합니다.
5.  **실행 (Action):** '실행 가능한 인사이트'가 생성되면(예: '대장암 전조증상 확률 7% 증가'), 알림 서비스를 통해 사용자 앱으로 즉시 푸시 알림을 보냅니다. 사용자는 앱의 대시보드에서 상세한 리포트와 권장 사항을 확인할 수 있습니다.

### **기술적 구현 시 예상되는 가장 큰 허들과 해결책**

**가장 큰 허들: 하드웨어의 신뢰성 및 개인 식별 문제 (Hardware Reliability & User Identification)**

- **문제점:**
  1.  **센서의 정확도 및 내구성:** 변기라는 고습도, 화학물질(세제)에 노출된 극한 환경에서 분광학/화학 센서가 수년간 일관된 정확도를 유지하기는 극도로 어렵습니다. 센서 오염 및 보정(Calibration) 문제가 치명적입니다.
  2.  **사용자 자동 식별:** 한 가정에 여러 구성원이 있을 때, 누가 변기를 사용했는지 자동으로, 100% 정확하게 구분하는 것은 매우 어려운 문제입니다. 체중, 사용 시간 패턴 등으로 추정할 수 있지만 오류 가능성이 존재하며, 데이터 오염은 전체 시스템의 신뢰도를 무너뜨립니다.

- **해결책 (Workaround & Phased Approach):**

  1.  **MVP 단계에서는 하드웨어 복잡도 최소화:** 처음부터 모든 것을 측정하는 '만능 센서'를 고집하지 않습니다.
      - **Workaround 1: 반자동 데이터 입력으로 시작.** 사용자가 시중의 소변 검사 스트립을 사용한 후, 변기에 부착된 간단한 '스캐너'에 보여주면 이를 인식하여 데이터를 디지털화하는 방식입니다. 이는 하드웨어 R&D 리스크를 최소화하고 소프트웨어와 데이터 플랫폼의 가치를 먼저 검증할 수 있는 탁월한 전략입니다.

### **MVP 데이터 흐름 (The Vision AI Pipeline)**

1.  **촬영 및 업로드 (Capture & Upload):** 사용자는 앱 내에서 지원되는 소변 검사 스트립 브랜드를 선택합니다. 앱은 해당 스트립에 맞는 촬영 가이드라인(네모난 박스 등)을 카메라 화면에 표시합니다. 사용자가 촬영하면, 이미지는 `user_id`, `strip_brand`, `timestamp` 메타데이터와 함께 S3 버킷으로 직접 업로드됩니다.
2.  **처리 요청 (Trigger):** S3에 새로운 이미지가 업로드되면, S3 Event Notification이 자동으로 SQS(Simple Queue Service)에 처리 요청 메시지를 보냅니다. 큐를 사용하는 이유는 갑작스러운 트래픽 증가에도 안정적으로 요청을 처리하기 위함입니다.
3.  **CV 분석 (Analyze):** CV 분석 서비스(Lambda)가 큐의 메시지를 받아 이미지 처리를 시작합니다.
    a. S3에서 원본 이미지를 다운로드합니다.
    b. OpenCV를 이용해 이미지의 기울기를 보정하고, 스트립 영역을 탐지합니다.
    c. 사전 정의된 스트립 브랜드의 레이아웃에 맞춰 각 바이오마커 패드 영역을 잘라냅니다.
    d. 각 패드의 평균 RGB/HSV 색상 값을 계산합니다.
    e. 해당 브랜드의 '색상-수치 변환표'를 참조하여 가장 근접한 색상의 수치 값(예: Glucose: 100mg/dL)으로 변환합니다.
4.  **저장 및 피드백 (Store & Feedback):** 분석된 최종 수치 데이터는 TimescaleDB에 시계열로 저장되고, 분석 과정의 메타데이터(원본 색상 값 등)는 Aurora에 저장됩니다. 데이터 저장이 완료되면, 앱으로 푸시 알림을 보내 사용자에게 분석이 완료되었음을 알립니다. 사용자는 앱을 열어 시각화된 차트와 리포트를 확인합니다.

### **MVP의 가장 큰 기술적 허들과 해결책: CV 정확도**

- **문제점:** **조명, 그림자, 카메라 성능** 등 통제되지 않는 환경 변수로 인해 스트립의 색상 값이 왜곡될 수 있습니다. 이는 서비스의 신뢰도와 직결되는 치명적인 문제입니다. "내 아이폰에서는 되는데, 아내의 갤럭시에서는 수치가 다르게 나와요"라는 불만은 서비스의 종말을 의미합니다.

- **해결책 (Robustness Engineering):**

  1.  **촬영 환경 강제 (Guided UI):** 단순히 카메라를 여는 것이 아니라, "흰색 배경 위에 스트립을 놓아주세요", "그림자가 생기지 않도록 촬영해주세요" 같은 가이드를 제공하고, 화면의 플래시를 강제로 켜서 광량을 일정하게 유지시킵니다.
