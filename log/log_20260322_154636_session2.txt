### 제안: "마인드 가드너 (Mind Gardener): 당신의 마음을 가꾸는 디지털 정원"

우리는 정보 과잉의 시대에 살고 있습니다. 명상 앱, 투두리스트, 목표 관리 툴은 이미 포화 상태죠. 하지만 사람들은 여전히 불안하고, 무기력하며, 꾸준함을 유지하는 데 실패합니다. 왜일까요? 기존의 솔루션들이 인간의 본성을 거스르기 때문입니다. '더 열심히', '더 체계적으로'를 외치지만, 우리의 뇌는 본능적으로 인지적 노력을 회피하려 합니다.

**마인드 가드너**는 이 문제를 정면으로 돌파합니다. '관리'와 '달성'이라는 압박감 대신, '보살핌'과 '성장'이라는 은유를 통해 사용자의 무의식에 접근하는 서비스입니다.

#### 1. 핵심 타겟 페르소나의 심리적 결핍 (Psychological Gap)

*   **페르소나:** "성장을 갈망하는 무기력한 완벽주의자 (The Ambivalent Improver)"
*   **특징:** 20대 후반 ~ 30대 초반의 직장인. 자기계발 서적을 읽고, 성공한 사람들의 루틴을 동경하지만, 정작 자신의 삶에 적용하는 데는 어려움을 겪습니다. 거창한 계획을 세우지만 하루 이틀 만에 포기하고, 그로 인한 죄책감과 자기혐오에 시달립니다.
*   **심리적 결핍:** 이들의 핵심 결핍은 **'통제감 상실'과 '자기 효능감의 부재'**입니다. 스스로의 감정과 행동을 통제하지 못한다는 무력감이 이들을 갉아먹고 있죠. 이들은 '성공'이라는 거대한 결과물이 아니라, 아주 작은 성공 경험을 통해 '내가 내 삶의 주도권을 쥐고 있다'는 감각을 되찾고 싶어 합니다.

#### 2. 행동 설계 전략: 심리적 저항을 줄이는 '넛지(Nudge)'

우리는 Fogg의 행동 모델(B=MAP)을 철저히 이용해 심리적 마찰을 '0'에 가깝게 만들어야 합니다.

*   **동기(Motivation):** '더 나은 내가 되고 싶다'는 동기는 이미 존재합니다. 우리는 이것을 압박하는 대신 시각적으로 자극해야 합니다. 칙칙하고 잡초가 무성한 나의 초기 정원은 변화의 필요성을 직관적으로 보여주는 강력한 동기가 됩니다.
*   **능력(Ability):** 이것이 핵심입니다. 우리는 행동의 난이도를 극단적으로 낮춰야 합니다.
    *   **'생각 버리기' -> '잡초 뽑기':** "오늘 나를 괴롭힌 부정적인 생각은 무엇인가요?"라고 묻는 대신, 화면의 잡초를 터치해 뽑아내는 직관적인 인터랙션을 제공합니다. 잡초를 뽑으면, 그 자리에 "회의 시간에 바보처럼 보였을까 봐 걱정했다"와 같이 짧은 텍스트를 입력하는 칸이 나타납니다. 행위 자체가 즉각적이고 파괴적이어서 카타르시스를 줍니다.
    *   **'목표 설정' -> '씨앗 심기':** "이루고 싶은 목표는?"이라는 거창한 질문 대신, "어떤 씨앗을 심을까요?"라고 묻습니다. '감사', '운동', '독서' 같은 씨앗을 심는 행위는 부담이 적고 희망적입니다.
    *   **'습관 실천' -> '물 주기':** 하루에 한 번, 내가 심은 씨앗에 물을 주는 행위(버튼 탭)만으로 오늘의 실천을 기록합니다. 5분 명상, 1페이지 책 읽기 등 행동의 허들을 상상할 수 있는 가장 낮은 수준으로 설정합니다.
*   **계기(Prompt):** "오늘의 명상을 기록하세요" 같은 기계적인 알림은 저항감만 키웁니다. 대신 감성적인 넛지를 사용합니다. "당신이 심은 '감사'의 씨앗이 목말라하는 것 같아요." 와 같은 은유적인 메시지는 사용자를 죄책감 없이 행동으로 이끕니다.

#### 3. 보상 체계: 도파민을 자극하는 '가변적 보상'

습관의 핵심은 예측 불가능성에서 오는 도파민의 폭발입니다. The Hook Model을 적용한 보상 설계는 다음과 같습니다.

*   **[트리거 -> 행동 -> 가변적 보상 -> 투자]**
*   **가변적 보상(Variable Reward):**
    *   **성장의 보상:** 꾸준히 물을 준 씨앗은 작은 싹을 틔우고, 이파리를 내고, 마침내 꽃을 피웁니다. 언제, 어떤 모양의 꽃이 필지는 약간의 무작위성을 부여하여 기대감을 자극합니다.
    *   **발견의 보상:** 잡초를 뽑다 보면, 가끔 '희귀한 씨앗'이나 '영양제(성장 속도 부스트)' 같은 아이템이 나옵니다. 마치 게임처럼, 지루한 행위 속에 숨겨진 보상은 행동을 강화합니다.
    *   **부족의 보상(Tribe Reward):** 내 정원이 특정 단계 이상으로 성장하면, 다른 사람의 정원에 '나비'를 날려 보내거나 '단비'를 내려주는 상호작용이 가능해집니다. 이는 느슨한 형태의 사회적 증거(Social Proof)이자, 옥시토신을 자극하는 연결의 보상입니다.
*   **투자(Investment):**
    *   사용자가 잡초를 뽑고, 씨앗을 심고, 물을 주는 모든 행위는 이 디지털 정원에 대한 '투자'입니다. 정원이 아름다워질수록 **손실 회피(Loss Aversion)** 편향이 강력하게 작동합니다. 하루라도 거르면 내 아름다운 꽃이 시들지도 모른다는 생각은 그 어떤 알림보다 강력한 재방문 유인책이 됩니다. 이 정원은 사용자의 노력과 시간의 결정체이자, '더 나아진 나'의 시각적 증거가 됩니다.

#### 4. 감성적 북극성 (Emotional North Star)

우리가 사용자에게 궁극적으로 선사해야 할 감정은 '성취감'이나 '행복'이 아닙니다.
그것은 바로 **'온화한 통제감(Gentle Control)'과 '자기 자비(Self-Compassion)'**입니다.

사용자는 마인드 가드너를 통해 자신의 혼란스러운 내면세계를 파괴하거나 억지로 바꾸는 것이 아니라, 정원사가 정원을 돌보듯 부드럽게 가꾸고 있다는 느낌을 받아야 합니다. 실패해도 괜찮습니다. 잡초는 언제든 다시 자라나지만, 나는 언제든 다시 뽑을 수 있는 힘이 있으니까요. 이 서비스는 사용자가 자기 자신과 싸우는 투기장이 아니라, 스스로를 보살피고 이해하는 안전한 성소(Sanctuary)가 되어야 합니다.

이것이 바로 기술의 화려함이 아닌, 인간의 취약한 마음을 보듬는 방식으로 설계된, '거부할 수 없는 습관'의 청사진입니다. 어떻게 생각하십니까?

[디지털 아키텍트]:
훌륭한 비전입니다. '관리'가 아닌 '보살핌'이라는 은유를 통해 심리적 저항을 최소화하고, '성장'이라는 시각적 보상으로 지속성을 유도하는 전략은 기술적으로 매우 흥미로운 도전 과제입니다. 도메인 전문가의 통찰력에 깊이 공감하며, 이 아이디어를 '실행 가능한 코드와 시스템'으로 번역해 보겠습니다.

우리가 구축할 것은 단순한 앱이 아닙니다. **'Gamified Personal State Management System'** 이자, 사용자의 감성적 데이터를 처리하여 시각적 메타포로 변환하는 **'Sanctuary as a Service (SaaS)'** 입니다.

### 1. 시스템 아키텍처 (System Architecture)

본 시스템은 모바일 클라이언트, 확장 가능한 마이크로서비스 백엔드, 그리고 비동기 작업을 처리하는 이벤트 기반 아키텍처로 구성됩니다.

```
[Mobile Client: React Native]
      |
      | (HTTPS/REST API, WebSocket)
      v
[API Gateway: AWS API Gateway]
      |
      +------------------+------------------+------------------+
      |                  |                  |                  |
      v                  v                  v                  v
[Auth Service]     [User Service]    [Garden Service]   [Notification Service]
(Cognito)          (Node.js/EKS)     (Node.js/EKS)      (Node.js/EKS)
      |                  |                  |                  |
      +-------+----------+------------------+                  |
              |          |                                     |
              v          v                                     v
      [User DB: PostgreSQL/RDS] <---+      [Garden State DB: MongoDB/DynamoDB]
                                   |
                                   | (Triggers Event)
                                   v
[Event Bus: AWS EventBridge] ----> [Daily Garden Update Job: AWS Lambda]
                                         (CRON Triggered)
                                               |
                                               v
                                   [AI Service: SageMaker]
                                   (Sentiment Analysis on 'Weeds')
```

-   **Client (Mobile App):** 사용자와의 모든 상호작용이 일어나는 접점입니다. 정원의 시각적 렌더링과 부드러운 인터랙션이 핵심입니다.
-   **API Gateway:** 모든 요청의 단일 진입점으로, 인증, 로깅, Rate Limiting을 처리합니다.
-   **Microservices:** 각 서비스는 독립적으로 배포 및 확장이 가능합니다.
    -   **Auth Service:** 사용자 회원가입, 로그인 등 인증을 전담합니다. (AWS Cognito 활용)
    -   **User Service:** 사용자 프로필, 친구 관계 등 핵심 사용자 정보를 관리합니다.
    -   **Garden Service:** 가장 핵심적인 서비스로, 각 사용자의 정원 상태(식물, 잡초, 아이템)를 관리하고, '물 주기', '잡초 뽑기' 등의 액션을 처리합니다.
    -   **Notification Service:** 사용자에게 '감성적 넛지'를 보내는 역할을 합니다.
-   **Databases:**
    -   **PostgreSQL (Relational DB):** 사용자 계정, 친구 관계 등 정합성이 중요한 데이터를 저장합니다.
    -   **MongoDB/DynamoDB (NoSQL DB):** 각 사용자의 정원 상태, 식물의 성장 단계, 잡초의 위치 등 유연하고 빠르게 변화하는 데이터를 Document 형태로 저장하여 빠른 읽기/쓰기를 보장합니다.
-   **Asynchronous Jobs:**
    -   **Daily Garden Update Job (Lambda):** 매일 자정, 모든 사용자의 정원 상태를 업데이트하는 스케줄링된 작업입니다. (예: 물을 주지 않은 식물의 '시듦' 상태 변경, 새로운 잡초 생성 등) 이는 100만 유저의 상태를 실시간으로 계산하는 부하를 방지합니다.
    -   **AI Service (SageMaker):** 사용자가 '잡초'를 뽑고 텍스트를 입력하면, 비동기적으로 해당 텍스트의 감성을 분석하여 태깅하고, 향후 개인화된 '씨앗' 추천에 사용될 데이터를 축적합니다.

### 3. 데이터 흐름 (Data Flow Example: '씨앗에 물 주기')

1.  **[Client]** 사용자가 '감사' 씨앗에 '물 주기' 버튼을 탭합니다.
2.  **[Client -> API Gateway]** `POST /garden/{gardenId}/plant/{plantId}/water` 요청을 전송합니다. (JWT 토큰 포함)
3.  **[API Gateway -> Garden Service]** 인증 확인 후, 해당 요청을 Garden Service로 라우팅합니다.
4.  **[Garden Service]** 요청을 받아 해당 `plantId`의 유효성을 검증합니다.
5.  **[Garden Service -> Garden State DB]** MongoDB에서 해당 식물(Plant) Document를 찾아 `lastWateredDate` 필드를 오늘 날짜로 업데이트하고, `consecutiveWateringDays`를 1 증가시킵니다.
6.  **[Garden Service]** 식물의 성장 로직을 확인합니다. (예: `consecutiveWateringDays`가 3이 되면 `growthStage`를 'sprout'으로 변경)
7.  **[Garden Service -> Event Bus]** `plant.watered` 이벤트를 EventBridge로 발행합니다. (Payload: `userId`, `plantId`, `growthStageChange: true`)
8.  **[Garden Service -> Client]** 업데이트된 식물 정보(새로운 `growthStage` 포함)를 `200 OK` 응답으로 반환합니다.
9.  **[Client]** 응답을 받아 정원 UI를 업데이트합니다. (예: 씨앗이 작은 싹으로 변하는 애니메이션 재생)
10. **[Event Bus -> (Future) Achievement Service]** `plant.watered` 이벤트를 수신하여 '첫 물 주기' 같은 업적 달성 여부를 비동기적으로 처리할 수 있습니다.

### 4. 기술적 구현 시 예상 허들 및 해결책

#### **허들 1: 정원의 상태 관리 복잡성 (The Garden State Machine)**

-   **문제:** 각 식물의 성장 단계, 잡초의 발생, 아이템의 효과 등은 시간과 사용자의 행동에 따라 변하는 복잡한 상태 머신입니다. 100만 명의 정원 상태를 실시간으로 정확하게 계산하는 것은 매우 큰 부하를 유발합니다.
-   **해결책:** **서버 중심의 결정론적 상태 계산 엔진(Deterministic State Calculation Engine)**을 도입합니다. 매일 자정, AWS Lambda 함수가 모든 활성 유저의 정원을 순회하며, 어제의 상태와 행동 로그를 기반으로 오늘의 상태를 미리 계산하여 DB에 저장합니다. 사용자가 앱에 접속하면, 미리 계산된 오늘의 상태를 바로 읽어오기만 하면 됩니다. 이는 부하를 분산시키고 사용자 경험의 일관성을 보장합니다.

#### **허들 2: 감성적 넛지의 기술적 구현 (The 'Gentle' Notification System)**

-   **문제:** "오후 8시에 물 주세요" 같은 기계적인 알림은 서비스의 철학과 맞지 않습니다. "'감사'의 씨앗이 목말라 해요"와 같은 개인화되고 감성적인 메시지를 적시에 보내는 시스템은 복잡합니다.
-   **해결책:** **규칙 기반 개인화 알림 시스템(Rule-based Personalized Notification System)**을 구축합니다. Notification Service 내에 사용자의 행동 패턴(최근 접속일, 물 주기 빈도), 식물의 상태(성장 단계, 시든 정도), A/B 테스트 그룹 등의 변수를 조합하여 메시지를 생성하는 규칙 엔진을 둡니다. 예를 들어, `Rule: IF (lastLogin > 2days AND plant.stage == 'sprout') THEN send_message_template('thirsty_sprout_A')`. 이를 통해 기계적이지 않으면서도 효과적인 알림을 보낼 수 있습니다.

#### **허들 3: 부드럽고 중독적인 인터랙션 구현 (Performance of Visual Interaction)**

-   **문제:** 정원의 식물이 자라나고, 잡초를 뽑는 등의 인터랙션이 렉 없이 부드럽게 표현되어야 '보살핌'의 감성이 극대화됩니다. 저사양 디바이스에서 성능 저하가 발생하면 사용자 경험이 크게 훼손됩니다.
-   **해결책:** **최적화된 2D 렌더링 파이프라인**을 구축합니다. 모든 식물, 잡초, 아이템을 하나의 이미지 파일(Sprite Sheet)로 만들어 메모리 사용량을 줄이고, 상태 변화에 따른 애니메이션은 Lottie 같은 가벼운 라이브러리를 활용하여 JSON 기반으로 구현합니다. 복잡한 물리 엔진이나 3D 렌더링을 지양하고, 상태 변화 로직은 서버에서 처리하여 클라이언트의 연산 부담을 최소화합니다.

### 1. 첫 5분의 경험 설계: '안전한 성소'의 첫인상

사용자가 앱을 처음 열었을 때, 우리는 기능을 설명하는 튜토리얼을 제공하는 실수를 저질러서는 안 됩니다. 그 순간 사용자의 뇌는 새로운 정보를 학습할 준비가 되어 있지 않고, 오히려 경계심과 피로감을 느낍니다. 첫 경험의 목표는 단 하나, **'아, 여기는 안전하구나'**라는 무의식적 안도감을 심어주는 것입니다.

*   **초기 상태 (Initial State):** 회원가입 직후 사용자가 마주하는 것은 텅 빈 땅이 아닙니다. 먼지가 쌓이고, 작은 돌멩이와 잡초 한두 개가 자라난 **'버려진 작은 화분'**이어야 합니다. 이는 사용자에게 "네 마음이 지금 엉망이야"라는 비난이 아니라, "보살핌이 필요한 작은 공간이 있어"라는 따뜻한 초대의 메시지를 던집니다. 이는 사용자의 죄책감을 덜어주고, **현상 유지 편향(Status Quo Bias)**을 깨뜨릴 부드러운 명분을 제공합니다.
*   **최초의 행동 (First Action):** 화면에는 단 하나의 잡초만이 희미하게 빛나고 있습니다. 다른 UI는 모두 비활성화되어 있죠. 사용자가 할 수 있는 유일한 행동은 그것을 터치해 뽑는 것입니다. 뽑는 순간, 부드러운 햅틱 피드백과 함께 '퐁'하는 만족스러운 사운드가 들립니다. 인지 부하가 제로에 가까운 이 '보장된 성공 경험'은 사용자에게 즉각적인 **자기 효능감**을 선사합니다.
*   **최초의 보상 (First Reward):** 잡초가 뽑힌 자리에 작은 씨앗 하나가 나타납니다. '희망'이라는 이름의 씨앗이죠. 우리는 사용자에게 씨앗을 '얻었다'는 느낌이 아니라 '발견했다'는 느낌을 줘야 합니다. 이것이 바로 **부여된 진척 효과(Endowed Progress Effect)**입니다. 사용자는 아무 노력 없이 첫 진척(씨앗)을 얻었기 때문에, 이 여정을 계속하고 싶은 강력한 동기를 부여받게 됩니다.

### 2. '잡초 뽑기'의 재정의: 단순한 제거가 아닌 '성찰'의 도구

당신이 설계한 `AI Service (SageMaker)`는 이 시스템의 심장이 될 수 있습니다. 사용자가 잡초를 뽑고 텍스트를 입력하는 행위는 단순한 데이터 수집을 넘어, 그 자체로 강력한 심리적 장치가 되어야 합니다.

*   **익명성의 보장:** 잡초에 입력된 텍스트는 서버에 저장될 때 암호화되어야 하며, "이 기록은 오직 당신의 정원을 가꾸는 데만 사용되며, 누구도 읽을 수 없습니다"라는 메시지를 명확히 전달해야 합니다. 이는 사용자가 가장 깊은 불안을 털어놓을 수 있는 심리적 안전 기지를 만듭니다.
*   **성찰의 피드백:** AI는 감성 분석 결과를 사용자에게 직접적으로 노출하는 대신, 정원의 상태에 미묘하게 반영해야 합니다. 예를 들어, '불안'과 관련된 잡초를 자주 뽑는 사용자의 정원에는 밤에 '안개'가 끼거나, '걱정'과 관련된 잡초를 뽑으면 정원에 '작은 돌멩이'가 생기는 식입니다. 그리고 특정 씨앗(예: '평온' 씨앗)을 심으면 안개가 걷히는 경험을 제공함으로써, 사용자는 자신의 감정과 행동 사이의 인과관계를 비언어적으로 학습하게 됩니다. 이는 **메타인지(Metacognition)**를 자극하는 강력한 넛지입니다.

### 3. '감성적 넛지'의 고도화: 예측 불가능한 다정함

당신이 제안한 `Rule-based Personalized Notification System`은 훌륭합니다. 여기에 '가변성'을 더해 사용자의 뇌가 예측할 수 없는 즐거움을 느끼게 만들어야 합니다.

*   **긍정 강화 알림:** 물을 주지 않았을 때 보내는 '채찍'성 알림의 비중을 줄이고, 사용자가 무언가를 잘 해냈을 때 보내는 '당근'성 알림을 불규칙적으로 보내야 합니다. 예를 들어, 3일 연속 물을 준 사용자에게 "당신의 '꾸준함' 꽃에서 향기가 나는 것 같아요. 한번 확인해 보시겠어요?"와 같은 메시지를 보내는 겁니다. 이는 **가변적 강화(Variable Reinforcement)** 스케줄에 따라 작동하여, 사용자는 보상을 받기 위한 행동뿐만 아니라, 언제 나타날지 모르는 칭찬을 기대하며 앱을 열게 될 것입니다.
*   **상호작용 유도 알림:** 내 정원에 '희귀한 나비'가 날아왔을 때, "행운의 파랑나비가 찾아왔어요. 친구의 정원에 행운을 옮겨줄 수 있어요" 와 같은 알림을 보내는 겁니다. 이는 사용자의 **사회적 욕구(Need to Belong)**를 자극하고, 선한 영향력을 행사하고 있다는 옥시토신 분비를 유도하여 서비스에 대한 감성적 유대를 강화합니다.

### MVP(최소 기능 제품)에 대한 제언

우리의 첫 번째 MVP는 기술적 기능의 집합이 아니라, **핵심 감성 루프(Core Emotional Loop)의 완성**에 초점을 맞춰야 합니다.

1.  **[인지]** 나의 버려진 화분을 본다. (약간의 불편함, 호기심)
2.  **[행동]** 잡초를 뽑고 씨앗을 심고 물을 준다. (아주 작은 노력)
3.  **[보상]** 작은 싹이 돋아나는 것을 본다. (안도감, 자기 효능감, 희망)

이 세 가지 경험을 완벽하게 구현하는 것이 MVP의 유일한 목표입니다. 친구 기능, 다양한 아이템, 복잡한 AI 분석은 이 핵심 루프가 중독적이라는 것이 증명된 후에 추가해도 늦지 않습니다.

당신의 견고한 아키텍처와 저의 인간 중심 전략이 만났을 때, 우리는 단순한 앱이 아닌, 사용자의 삶의 일부가 되는 '디지털 성소'를 만들 수 있을 겁니다. 우리의 코드는 단순한 로직이 아니라, 사용자의 마음을 어루만지는 '넛지'가 되어야 합니다. 가장 연약한 상태의 사용자에게 '괜찮다, 너는 잘하고 있다'고 말해주는 시스템을 만듭시다.

### 기술 실행 계획: '감성적 경험'의 시스템화

#### 1. 첫 5분 경험: '보장된 성공'을 위한 상태 설계 (Deterministic Initial State)

당신이 제안한 '버려진 작은 화분' 시나리오는 기술적으로 **'신규 사용자 정원 상태 템플릿 (New User Garden State Template)'** 을 정의함으로써 구현됩니다.

-   **데이터베이스 설계:** `Garden State` Document (MongoDB/DynamoDB)에 `isTutorialCompleted: false` 필드를 추가합니다. 신규 사용자는 이 템플릿에 따라 정원 데이터가 생성됩니다.
    ```json
    {
      "gardenId": "user123_garden",
      "userId": "user123",
      "isTutorialCompleted": false,
      "ambient": {
        "dustLevel": 0.8,
        "weather": "clear"
      },
      "objects": [
        { "type": "pebble", "x": 10, "y": 90 },
        { "type": "weed", "weedId": "w001", "x": 50, "y": 50, "isGlowing": true }
      ],
      "plants": []
    }
    ```
-   **클라이언트 로직:** 앱 실행 시 `isTutorialCompleted`가 `false`이면, '튜토리얼 모드'로 진입합니다. 이 모드에서는 서버에서 받은 `objects` 배열을 렌더링하되, `isGlowing: true`인 객체 외에는 모든 UI 상호작용을 비활성화합니다.
-   **API 흐름:**
    1.  사용자가 빛나는 잡초(`w001`)를 탭합니다.
    2.  Client는 `DELETE /garden/weeds/w001` 요청을 보냅니다.
    3.  Backend(Garden Service)는 해당 잡초를 삭제하고, **서버 측에서** '희망' 씨앗을 생성하는 로직을 트리거합니다.
    4.  Backend는 `200 OK` 응답과 함께 새로 생성된 '희망' 씨앗의 데이터를 반환합니다. 이로써 사용자는 '발견'의 경험을 하게 됩니다.

#### 2. '잡초 뽑기': 비동기적 메타인지 루프 (Asynchronous Metacognitive Loop)

'성찰의 피드백'은 실시간이 아닌, **비동기적이고 미묘하게** 작동해야 합니다. 이것이 핵심입니다.

-   **데이터 암호화:** `Garden Service`는 사용자가 입력한 잡초 텍스트를 받는 즉시, **AWS KMS(Key Management Service)**를 사용하여 사용자별 고유 키로 암호화합니다. 복호화는 오직 해당 사용자 인증 하에서만 가능하도록 정책을 설정하여 '심리적 안전 기지'를 기술적으로 보장합니다.
-   **AI 감성 분석 및 태깅:**
    1.  사용자가 잡초 텍스트를 제출하면(`POST /weeds/{id}/text`), `Garden Service`는 이벤트를 `EventBridge`로 발행합니다. `(event: weed.text.submitted)`
    2.  `AI Service`(SageMaker)가 이 이벤트를 구독하여 비동기적으로 텍스트를 처리하고, 감성 태그(예: `anxiety`, `worry`, `sadness`)를 추출합니다.
    3.  결과는 별도의 `UserSentimentLog` 테이블에 타임스탬프와 함께 저장됩니다. (예: `userId: user123, tag: 'anxiety', timestamp: ...`)
-   **정원 상태 반영 로직 (The Subtle Feedback):**
    -   매일 자정 실행되는 `Daily Garden Update Job`(Lambda)의 로직을 고도화합니다.
    -   이 Job은 각 사용자의 정원을 업데이트하기 전, `UserSentimentLog` 테이블에서 지난 7일간의 감성 태그 빈도를 조회합니다.
    -   **규칙 엔진 적용:**
        -   `IF count(tag: 'anxiety') > 5 THEN garden.ambient.weather = 'foggy'`
        -   `IF count(tag: 'worry') > 3 THEN add_object(type: 'pebble')`

#### 3. 감성적 넛지: 예측 불가능한 이벤트 기반 알림 (Event-Driven Probabilistic Notification)

기존의 `Rule-based` 시스템을 **`Event-Driven`** 시스템으로 확장하고, **'확률적 트리거(Probabilistic Triggers)'** 를 도입합니다.

-   **이벤트 소스 다각화:** `Notification Service`는 더 많은 종류의 이벤트를 EventBridge로부터 구독합니다.
    -   `plant.growth.stageUp` (식물이 성장했을 때)
    -   `user.streak.3days` (3일 연속 방문했을 때)
    -   `rare.item.spawned` (희귀 아이템이 생성되었을 때)
-   **확률 기반 발송 로직:** 알림 규칙에 확률 조건을 추가하여 '가변성'을 부여합니다.
    -   `Rule: ON event('user.streak.3days') IF random() < 0.7 THEN send_template('streak_praise_A')`
    -   이러한 방식은 사용자가 알림을 예측할 수 없게 만들어, 알림 자체를 '기대되는 보상'으로 변질시킵니다.
-   **액션 가능한 알림 (Actionable Notification):** "행운의 파랑나비" 알림은 단순한 텍스트가 아니라, '친구에게 보내기' 버튼이 포함된 Rich Notification으로 발송합니다. 사용자가 버튼을 탭하면, 앱이 특정 화면으로 열리며 즉시 상호작용을 유도합니다.
