### 1. 기술적 관점의 재해석 (Re-interpretation)

'Nexus Build'는 본질적으로 **"BIM(Building Information Modeling)을 중심으로 한 Event-Driven 실시간 협업 플랫폼"**입니다. 모든 상호작용(설계 변경, 자재 입고, 작업 보고, 이슈 발생)을 '이벤트(Event)'로 간주하고, 이 이벤트를 모든 이해관계자에게 실시간으로 전파하여 '단일 진실 공급원(Single Source of Truth)' 역할을 하는 BIM 모델에 동기화하는 시스템입니다.

-   **Digital Twin:** 정적 3D 모델이 아닌, 현실 세계의 상태(State)와 이벤트(Event)를 실시간으로 반영하는 '살아있는' 디지털 모델입니다.
-   **OS (Operating System):** 개별 기능(설계, 시공, 감리)을 제공하는 앱(App)들이 Nexus Build라는 OS 위에서 상호작용하는 생태계를 의미합니다. 이를 위해선 강력한 API가 필수적입니다.

### 2. 시스템 아키텍처 다이어그램 (System Architecture)

본 시스템은 확장성과 데이터 처리량을 극대화하기 위해 MSA(Microservices Architecture)를 기반으로 설계합니다.

```
[Clients] <--> [API Gateway] <--> [Backend Microservices] <--> [Data Stores & Processing]

[Clients]
  - Web App (React/Vue.js + Three.js/APS Viewer) -> 설계자, 관리자, PM
  - Mobile App (React Native/Flutter) -> 현장 관리자, 인부, 감리
  - 3rd Party API (REST/GraphQL) -> ERP, 설계 툴 플러그인

      |
      v (HTTPS, WebSocket)
      |

[API Gateway (e.g., AWS API Gateway, Kong)]
  - 인증/인가 (Authentication/Authorization)
  - 요청 라우팅 (Request Routing)
  - Rate Limiting

      |
      v (gRPC/REST)
      |

[Backend Microservices (Docker Containers on Kubernetes)]
  - User Service: 사용자 계정, 권한 관리
  - Project Service: 프로젝트, 예산, 일정 정보 관리
  - BIM Service: BIM 모델(IFC/RVT) 업로드, 파싱, 경량화, 뷰어 데이터 변환 (Heavy Lifting)
  - Real-time Service: WebSocket 연결 관리, 실시간 이벤트 전파 (Socket.IO, Centrifugo)
  - Issue Tracking Service: 현장 이슈, 사진, 코멘트 관리
  - Marketplace Service: 자재/장비/인력 거래 로직
  - Analytics Service: 데이터 집계 및 리포트 생성

      |
      v (TCP/IP, AMQP)
      |

[Data Stores & Processing]
  - [Relational DB (PostgreSQL + PostGIS)]
    - User, Project, Issue 등 정형 데이터 저장
    - PostGIS: 지리/공간 정보 기반 쿼리 (e.g., 특정 좌표 5m 반경 내 이슈 검색)
  - [Object Storage (AWS S3 / GCS)]
    - BIM 원본 파일, 도면, 현장 사진/동영상, 드론 데이터 등 대용량 파일 저장
  - [Cache (Redis)]
    - 실시간 세션 정보, 자주 조회되는 데이터 캐싱
    - Pub/Sub 기능으로 Real-time Service의 수평 확장 지원
  - [Message Queue (RabbitMQ / AWS SQS)]
    - BIM 파일 변환 등 오래 걸리는 작업을 비동기 처리하기 위한 통로
  - [Data Warehouse (BigQuery / Redshift)]
    - 모든 서비스의 로그 및 이벤트 데이터를 수집/저장하여 AI 모델 학습 및 인사이트 추출
```

### 4. 데이터 흐름 (Data Flow Example: 현장 이슈 발생 시)

1.  **[Mobile App]** 현장 인부 '김씨'가 모바일 앱으로 BIM 모델의 특정 벽체(객체 ID: `wall_123`)를 클릭하고, '균열 발생'이라는 제목과 함께 사진을 찍어 이슈를 등록합니다.
2.  **[API Gateway]** 모바일 앱의 요청(POST `/projects/proj_A/issues`)은 API Gateway를 통해 인증 처리 후 'Issue Tracking Service'로 라우팅됩니다. 사진 파일은 S3에 직접 업로드(Presigned-URL 방식)됩니다.
3.  **[Backend]**
    *   'Issue Tracking Service'는 이슈 정보(내용, 작성자, `wall_123` 연동 정보, S3 사진 URL)를 PostgreSQL에 저장합니다.
    *   저장 후, 'Issue Created' 이벤트를 Message Queue(RabbitMQ)에 발행(Publish)합니다.
4.  **[Real-time & Notification]**
    *   'Real-time Service'는 Message Queue로부터 이벤트를 구독(Subscribe)하여, 해당 프로젝트 채널에 접속 중인 모든 클라이언트(설계자, 감리 등)에게 WebSocket을 통해 실시간으로 푸시합니다.
    *   별도의 Notification Service가 이벤트를 받아 관련자들에게 이메일/푸시 알림을 보냅니다.
5.  **[Web App]** 설계자 '박씨'의 웹 브라우저 화면에서 알림이 뜨고, BIM 뷰어의 `wall_123` 객체 위에 새로운 이슈 아이콘이 실시간으로 생성됩니다. 아이콘을 클릭하면 김씨가 올린 사진과 내용을 즉시 확인할 수 있습니다.

### 5. 기술적 구현 시 예상되는 가장 큰 허들과 해결책

1.  **허들: BIM 데이터 처리의 복잡성과 성능**
    *   **문제:** BIM 파일(RVT, IFC 등)은 수 GB에 달하며, 구조가 매우 복잡합니다. 이를 웹/모바일에서 실시간으로 렌더링하는 것은 엄청난 기술적 난제입니다. 자체 파서와 뷰어를 개발하는 것은 수년이 걸릴 수 있습니다.
    *   **해결책:** **Autodesk Platform Services (APS) 또는 Speckle과 같은 전문 PaaS/오픈소스 활용.** 이 서비스들은 복잡한 BIM 파일을 업로드하면 알아서 웹에서 스트리밍 가능한 경량 포맷(SVF, glTF)으로 변환해주고, 강력한 뷰어 API를 제공합니다. 우리는 이 API를 활용하여 비즈니스 로직(이슈 연동, 데이터 시각화) 구현에만 집중합니다. 초기에는 비용이 발생하지만, Time-to-Market을 1년 이상 단축시킬 수 있습니다.

2.  **허들: 불안정한 현장 네트워크와 오프라인 지원**
    *   **문제:** 건설 현장은 Wi-Fi나 5G 연결이 불안정한 경우가 많습니다. 온라인 상태에서만 작동하는 앱은 현장에서 무용지물입니다.
    *   **해결책:** **모바일 앱의 오프라인 우선(Offline-First) 설계.**
        *   **로컬 DB:** 모바일 기기에 SQLite 같은 로컬 DB를 탑재하여 프로젝트 데이터(경량화된 모델, 이슈 등)를 저장합니다.
        *   **동기화 엔진:** 네트워크 연결이 복구되면 로컬 DB의 변경사항(생성/수정된 이슈)을 서버와 동기화하는 로직을 구현합니다. 데이터 충돌(Conflict)을 해결하기 위한 정책(e.g., Last-Write-Wins) 수립이 중요합니다.

3.  **허들: 데이터의 상호운용성(Interoperability)**
    *   **문제:** 건설 산업은 수많은 종류의 소프트웨어(설계, 예산, 일정 관리 등)가 파편화되어 사용됩니다. Nexus Build가 이들과 연동되지 않으면 '중앙신경계'가 될 수 없습니다.
    *   **해결책:** **API 우선주의(API-First) 및 플러그인 아키텍처 설계.**
        *   초기부터 잘 설계된 REST/GraphQL API를 외부에 공개하여 다른 시스템이 Nexus Build의 데이터에 접근하고 이벤트를 발생시킬 수 있도록 합니다.
        *   Autodesk Revit, Navisworks 등 주요 설계 툴을 위한 플러그인을 직접 개발하여, 설계자가 자신의 툴 안에서 Nexus Build와 실시간으로 동기화할 수 있는 경험을 제공합니다.

### MVP Scope 제안

6개월 내 시장 검증을 위한 MVP는 다음 핵심 기능에 집중해야 합니다.

1.  **프로젝트 생성 및 팀원 초대 기능**
